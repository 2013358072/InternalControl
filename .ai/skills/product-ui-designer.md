# 产品 UI 设计规范

## 先输出 UI 规格

新页面编码前，先整理一份简洁的 UI 规格：

- 目标用户和核心任务
- 页面布局区域
- 主要用户路径
- 需要使用的组件
- UI 需要的数据
- loading、empty、error、permission 等状态
- 主题方向，用 CSS 变量描述，不要给硬编码颜色
- 若现有主题 token 不足，先定义需要新增的品牌/材质/背景/图表 token，再进入实现

## 产品页面默认策略

对于 SaaS、后台、CRM、工作流类产品：

- 优先使用“筛选栏 + 表格/列表 + 详情抽屉/弹窗”的结构。
- 标题保持紧凑，信息展示要便于扫读。
- 卡片只用于重复实体、摘要指标或确实需要框住的工具区域。
- 高频操作优先用表格操作列、分段 tabs、状态标签、侧边抽屉承载。
- 除非用户明确要求落地页，否则不要做装饰性强的营销页结构。

## 实现交接要求

UI 规格必须能直接映射到本模板：

- 功能型控件优先来自 `@/components`
- 展示型模块可使用页面局部组件，以保证设计表达
- 页面结构放在 `src/pages/<模块名>/`
- 主题变量来自 `src/styles/themes/`
- API 调用放到 `src/api/`
- 只要涉及与后端交互的数据场景（列表、详情、表单提交、状态流转、权限、统计等），先定义 `src/mocks/` 数据并让页面可基于 mock 运行；没有后端时不得把数据直接散落在页面 JSX 中。

## 设计质量检查

- 主操作清晰可见，并使用动词命名。
- 页面视觉不应被现有表单/后台组件压平；品牌化或展示型区域要有明确的版式和视觉记忆点。
- 筛选条件可以重置。
- 表格或列表有空状态和加载状态。
- 表单有禁用态和提交中状态。
- 危险操作需要确认。
- 状态表达不能只依赖颜色。
- 移动端有明确的纵向堆叠顺序。
